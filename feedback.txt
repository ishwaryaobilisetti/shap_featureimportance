. Why did you choose XGBoost for this interpretability task? What were its advantages?

I chose XGBoost for three key reasons: First, its gradient boosting architecture delivers strong predictive performance (achieved AUC 0.8312), which is essential since interpretability is only valuable when the underlying model is accurate. Second, XGBoost integrates seamlessly with SHAP's TreeExplainer, which computes exact SHAP values in polynomial time rather than approximations—this matters because approximate methods can misattribute feature contributions. Third, XGBoost handles class imbalance elegantly through scale_pos_weight, crucial for our 73%/27% churn distribution. Unlike deep learning models requiring computationally expensive KernelSHAP, tree-based models allow us to compute explanations for thousands of predictions efficiently while maintaining theoretical guarantees from game theory.

2. What was the most surprising or counter-intuitive insight you discovered from the SHAP analysis?

The most surprising finding was the interaction between tenure and contract type in the dependence plots. I expected tenure alone to be a strong predictor, but SHAP revealed that low-tenure customers on month-to-month contracts had dramatically higher churn risk than low-tenure customers on yearly contracts. This means the "honeymoon period" risk isn't universal—it's conditional on commitment level. Additionally, I found that customers without OnlineSecurity and TechSupport had higher SHAP values pushing toward churn, suggesting these aren't just add-on revenue—they're retention anchors that create switching costs and perceived value.

3. Explain the difference between global and local interpretability. How did SHAP help you address both?

Global interpretability answers "what features matter overall across all predictions?" I addressed this using:

Beeswarm plot: Shows feature importance ranking AND directional impact (e.g., high monthly charges → churn)
Bar chart: Aggregated mean |SHAP| values for clear ranking
Local interpretability answers "why did the model make THIS specific prediction?" I addressed this using:

Force plots: Visualize how each feature pushes an individual prediction from the base value
Waterfall plots: Show cumulative feature contributions building to the final output
SHAP uniquely bridges both levels because the same underlying SHAP values can be aggregated for global insights or examined individually for local explanations—they're mathematically consistent across scales, unlike other methods like LIME which only provide local approximations.

4. How did SHAP feature importances compare to native importance? If they differed, why?

The rankings were largely consistent for top features (Contract, Tenure, MonthlyCharges), validating the model's behavior. However, notable differences emerged: TotalCharges ranked higher in native gain importance but lower in SHAP. This occurs because gain-based importance measures how often a feature is used in splits and how much it improves predictions at those splits—it's a measure of usage frequency. SHAP measures actual impact on predictions regardless of split count. A feature can be used in many small splits (high gain) but have modest actual impact on outputs (lower SHAP). Additionally, gain-based importance doesn't account for feature interactions as accurately—SHAP's Shapley value foundation properly distributes credit among correlated features.

5. Based on your SHAP analysis, what are two concrete business recommendations?

Recommendation 1: Implement a "Contract Upgrade" Retention Program
Since Contract type was the #1 SHAP feature (mean |SHAP| = 0.899), proactively target month-to-month customers at the 6-9 month mark with incentivized upgrades (10-15% discount for 1-year commitment). The dependence plot shows this single change dramatically reduces churn probability. Expected ROI: High-risk segment directly addressed.

Recommendation 2: Create 90-Day Enhanced Onboarding
The tenure dependence plot revealed a critical "danger zone" in the first 12 months where SHAP values are strongly positive (pushing toward churn). Implement milestone check-ins at days 30, 60, and 90 with proactive support outreach and service education. Additionally, bundle OnlineSecurity as a free trial for new customers—SHAP showed customers without this service have significantly higher churn propensity.

6. What technical challenges did you face implementing SHAP, and how did you overcome them?

Two main challenges arose:

Challenge 1: Expected Value Array Handling
Newer SHAP versions return explainer.expected_value as an array for binary classification (one value per class), not a scalar. This caused errors in force plots. I resolved this by extracting the positive class value: expected_value = float(expected_value[1]) if len(expected_value) > 1 else float(expected_value[0]).

Challenge 2: SHAP Values List Format
Similarly, shap_values() returns a list of two arrays for binary classification. I standardized this by checking: if isinstance(shap_values_raw, list): shap_values = shap_values_raw[1] to consistently use positive-class SHAP values across all visualizations.

These weren't documented clearly, so debugging required understanding SHAP's internal handling of multi-class outputs and ensuring consistency across summary plots, force plots, and waterfall plots.
